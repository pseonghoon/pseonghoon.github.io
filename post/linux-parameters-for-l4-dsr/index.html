L4 DSR과 arp_ignore, arp_announce

Sep 27, 2020 linux tcp/ip
Share on:

L4 load balancer를 DSR(Direct Server Return) 모드로 구성할 때, 대상 Linux host의 kernel parameter로 arp_ignore,arp_announce를 설정한다. load balancer VIP에 대한 ARP request 에 서버가 응답하지 않게 하기 위해서다.

이 parameter들을 설정하는 방법에 대해 잘못된 정보가 널리 알려져 있는데, 그 정확한 의미를 알아보자

설정 방법 요약

DSR 로 L4 virtual server를 설정할 때 real server 의 loopback interface에 virtual server의 VIP를 추가해야 한다.

VIP의 주인은 real server가 아니라 L4 스위치이지만 DSR 모드로 L4가 real server에 forwarding한 패킷의 destination IP 가 VIP이기 때문에 이 것을 real server가 수신하게 하기 위한 것이다.

loopback interface 에 VIP를 추가한 후에는 arp_ignore, arp_announce 이 두 가지 kernel parameter도 적절히 설정을 해줘야한다.

이 parameter 들의 이름은 net.ipv4.conf.<target interface|all|default>.arp_ignore , net.ipv4.conf.<target interface|all|default>.arp_announce 의 형태이고 /etc/sysyctl.conf 을 아래처럼 설정하는 것을 권장한다.

1#/etc/sysctl.conf 
2net.ipv4.conf.default.arp_ignore = 1
3net.ipv4.conf.default.arp_announce = 2
4
5#loopback interface 의 설정은 할 필요가 없지만 아래처럼 하는 것이 명확하다
6net.ipv4.conf.lo.arp_ignore = 0 
7net.ipv4.conf.lo.arp_announce = 0
  • /etc/rc.local 에 스크립트로 설정하는 방식을 쓰면 안되고 /etc/sysctl.conf에 설정해야 한다. /etc/rc.local은 시스템 초기화 과정의 후반부에 실행되기 때문에 interface 가 올라오면서 Gratuitous ARP 가 전송 될 때는 해당 설정이 반영되지 않는다.
  • ‘default’ 대신 ‘all’을 써도 원하는 결과를 얻을 수 있으나 all로 설정하면 개별 interface마다 다른 설정을 넣을 수가 없으니 default가 더 유연하다
  • ‘default’ 와 ‘all’ 의 차이는 아래와 같다 . . http://unix.stackexchange.com/questions/90443/sysctl-proc-sys-net-ipv46-conf-whats-the-difference-between-all-defau요약하면 default는 interace가 처음 생성될 때 copy되는 값이고 all은 모든 interace에 영향을 끼치는 값이다. all 에 설정된 값과 interface별로 설정된 값 중에 실제로 어떤 값이 효력을 가질지는 parameter 별로 다르다.

arg_ignore

ARP request 를 받았을 때 응답을 줄지 여부를 결정한다.

  • L4의 VIP를 real server 의 loopback interface 에 추가했을 때, real server가 Linux라면 default 동작은 이 VIP에 대한 ARP request에 응답을 하는 것이다
  • real server에서 VIP에 대한 ARP request 를 받았을 때 응답을 하면 안되므로 arp_ignore 설정을 default 값인 0이 아닌 1로 해야 한다.
  • arp_ignore는 interface 별로 설정할 수 있는데, loopback interface에다 설정하는 것이 아니라 실제로 통신이 일어나는 인터페이스에 설정하는 것이다

설정값의 의미

  • 0 (default) : ARP request 가 도착했을 때, 조회 대상 IP가 이 시스템의 어떤 interface 든 (loopback 포함) 설정되어 있기만 하면 응답함.
  • 1 : arp_ignore가 1로 설정된 interface를 통해 ARP request가 도착하면 조회 대상 IP가 이 interface에 설정되어 있을 때만 응답을 보냄.
    ARP request는 loopback이 아니라 실제 통신이 일어나는 interface에 도착하므로 실제 interface에서 arp_ignore를 1로 설정해 줘야한다.lookback에는 ARP 패킷 등의 외부 패킷이 도달하지 않으므로 이 설정으로 real server가 VIP에 대한 ARP request에 응답을 하는 동작을 방지할 수 있다.

arp_announce

ARP request를 보낼 때 ‘sender IP 주소’를 어떻게 설정할지를 정하는 parameter. 여기서 sender IP 주소는 IP header 의 src IP가 아니라 ARP request 내용 (payload) 의 일부분임 . ARP는 L2 (Ethernet 등)로 통신하는 protocol 이기 때문에 IP헤더가 없다.

ARP 패킷의 구조

설정값의 의미

  • 0 (default) : 시스템에 설정된 어떤 IP든 사용할 수 있음.
  • 2 : ARP request 를 발생하게 한 IP packet (이 패킷의 dst IP 에 해당하는 MAC 주소를 알기 위해 ARP request 가 전송됨) 의 src IP를 무시하고 실제로 ARP request가 전송되는 interface에 설정된 IP를 가능하면 사용 . (실제 기준은 좀 더 복잡하니 문서의 초반에 공유된 링크를 참고 )

이 파라미터를 사용하는 목적은 아래와 같다.

호스트나 네트워크 장비의 ARP cache가 업데이트되는 것은 기본적으로 자신이 보낸 ARP request를 받은 IP주인이 보낸 ARP reply를 받았을 때이지만, ARP request를 받았을 때도 패킷의 내용 (payload) 에 포함된 sender IP 주소와 sender MAC 주소를 사용해서 ARP cache를 업데이트한다. 따라서 real server에서 ARP request를 보낼 때 sender IP 값이 VIP로 되어있으면 이 패킷을 받는 서버들이 자신의 ARP cache 에 VIP에 해당하는 MAC 주소를 real server의 것으로 업데이트하게 되므로 그 것을 방지하기 위한 설정이다.

arp_ignore 가 0으로 설정된 real server의 동작을 확인하는 테스트 방법

loopback interface 에 설정한 VIP 에 대해 ARP relay를 전송하는 현상을 확인하는 방법이다. host A , B 모두 eth0 으로 10.41.66.0/24 네트웍으로 통신하고 있을 때 아래처럼 확인할 수 있다. host A 가 real server 역할이고 10.41.66.100 이 VIP인 셈이다.

host A

1$ sudo sysctl -w net.ipv4.conf.eth0.arp_ignore = 0  
2$ sudo ifconfig lo:0 10.41.66.100 netmask 255.255.255.255

host B

1$ sudo arping -I eth0 -b 10.41.66.100
2ARPING 10.41.66.100 from 10.41.66.45 eth0
3Unicast reply from 10.41.66.100 [00:22:19:BA:D1:DF]  0.641ms   => 00:22:19:BA:D1:DF 은  host A 의 interface의 MAC 주소
4Unicast reply from 10.41.66.100 [00:22:19:BA:D1:DF]  0.609ms
5Unicast reply from 10.41.66.100 [00:22:19:BA:D1:DF]  0.614ms

위의 결과에서 보다시피 host A 는 lookback에 설정된 10.41.66.100 에 대한 ARP request를 받았을 때도 자신의 MAC 주소로 응답한다.
